# 建立遊戲實作總計畫

請讀取並分析：

- @遊戲設計文件
- @Storyboard
- AGENTS.md
- 既有程式、場景、Prefab 與相關文件

建立「Game Implementation Roadmap」，將遊戲拆成可逐一實作、整合及驗收的功能切片。

本階段僅能唯讀分析與撰寫 Roadmap。在我核准前，不得修改程式碼或 Unity 資產、建立詳細 Spec／Plan，或開始實作。

遵守 AGENTS.md；若它與本提示詞的階段邊界衝突，將問題列入 Open Questions 並停止相關工作。

## 規劃規則

先整理專案現況、既有功能、工具能力與限制。不得只因找到相關腳本，就認定功能已完成。

每個功能切片必須：

- 產生單一、玩家可見或可操作的結果。
- 能在 Unity 中獨立整合與驗收。
- 列出相依切片。
- 區分 AI 工作與使用者 Unity 工作。
- 定義 AI 與人類驗收條件。

純技術工作不單獨成為切片，應放入 AI Responsibility。

AI 無法操作的 GameObject、Prefab、Component、腳本掛載、Inspector、Layer、Tag、場景及視覺設定，放入 Human Unity Work。

每個切片的人工整合應盡量在約 15 分鐘內完成；超過或包含多個獨立結果時應考慮拆分。

依相依性排序，並優先形成最小可遊玩核心流程。不得擅自補充重要遊戲規則；缺漏、衝突與歧義列入 Open Questions。

## Roadmap 狀態

每個切片只能使用：

- Pending：Spec 尚未核准。
- Existing — Unverified：已有相關實作，但尚未完整驗證。
- Spec Approved：Spec 已核准。
- Plan Approved：Plan 已核准，可以實作。
- In Progress：正在實作或進行 AI 驗證。
- Ready for User Integration：AI 工作完成，等待 Unity 整合及人工驗收。
- Accepted：AI 與人類驗收皆已通過。
- Blocked：因未決設計、必要資源或外部條件無法繼續。

正常流程：

Pending
→ Spec Approved
→ Plan Approved
→ In Progress
→ Ready for User Integration
→ Accepted

新功能初始為 Pending。正常尚未完成的 Depends On 不算 Blocked。狀態改變時同步更新 Status Note 與 Last Updated。

## 驗收邊界

AI Acceptance 只能包含能取得客觀證據的項目，例如靜態分析、實際執行的編譯或測試、架構及介面檢查。

畫面、手感、動畫、碰撞、物理及 Play Mode 行為列入 Human Unity Acceptance，除非 AI 確實透過工具執行並取得證據。

未實際驗證的項目不得宣稱通過。

## 文件產出

以中文寫入：

docs/roadmaps/YYYY-MM-DD/game-implementation-roadmap.md

若專案已有文件慣例，遵循既有慣例並回報路徑。

文件包含：

- Status: Draft
- Current Project State
- Source Documents
- Project Overview
- Assumptions
- Open Questions
- Implementation Strategy
- Roadmap Status Definitions
- Feature Slices
- Milestones
- Definition of Done
- Risks

Feature Slices 必須依建議實作順序列出目前範圍內的全部切片。

完成後用中文回報：

1. 切片總數。
2. 最小可遊玩流程與第一個里程碑。
3. 人工工作較多或需要拆分的切片。
4. 需要我決定的問題。
5. 既有功能及工具限制。

完成後停止，等待我核准。
