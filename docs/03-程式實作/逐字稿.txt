13頁
spec 其實就是規格文件

舉例來說，原本你跟 ai 討論功能的紀錄，是不會保留對

這些資料會隨著你跟 ai 的對話，而逐漸遺失，因為 context 會把舊資料逐漸移出

而 spec 的做法是，把這些資料保留起來，並整理成一份文件

所以 spec 就是被壓縮過的 context

有了這份 context，之後要擴充或維護什麼功能，都有個依據

你就不用請 ai 在掃秒整個專案，浪費多餘的token

14頁
spec 文件的結構，其實沒有固定的格式

大家可以寫自己覺得重要的資訊就好

那我們這邊就列出，幾個我們覺得重要的欄位給大家參考

user story 我們覺得滿重要的，因為這是ai可以快速理解功能需求的欄位

input/output 就是輸入輸出，這個欄位對遊戲滿重要的

rules 就是功能的規則長怎樣，例如

non-goals 這個比較特別，這個是不做什麼的欄位，在ai時代下，要做什麼都可以，但在一個功能中做太多事情，反而會讓功能變得更難維護或擴充功能，所以定義不做什麼，反而是ai時代下更需要做的事情

acceptance criteria 這個也滿重要的，這個欄位是驗證功能的意思，怎麼定義這個功能要怎麼驗收，怎樣驗收才算打標，我覺得滿重要的

notes 這個欄位就是指，功能在設計上，可能有做特殊設計，就可以寫在這過欄位
