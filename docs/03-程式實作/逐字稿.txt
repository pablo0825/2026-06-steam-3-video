S14-2

S15
P1
spec 其實沒有固定的格式

所以我們這邊，就列出幾個欄位，給大家參考

在這些欄位中，

有兩個欄位，想跟大家特別提一下，

分別是 non-goals 和 acceptance criteria 


--------
P1
第一個欄位是，user Story，

這是前面有提過的

我們覺得這個欄位，還滿重要的，

因為這是 ai 可以快速理解，你想要的功能需求的欄位

P2
第二個是，input 和 output，

這個欄位，就是定義玩家要輸入什麼，以及遊戲要輸出什麼

例如，玩家按下空白鍵, 遊戲中的角色就跳一下

我們需要去確認清楚，玩家的輸入和遊戲的輸出是甚麼

P3
第三個是，rules 

這個欄位，就是定義功能是怎麼運作的

例如，你的角色踩到尖刺，是會受傷扣血，還是直接死亡，

這個就需要事前，先寫清楚
-----


P4
首先，non-goals 的部分

這個滿特別的，這是不做什麼功能的欄位，

在 ai 時代下，要做什麼，其實都很方便，

但在一個功能中，做太多事情，

會讓功能變得很難維護，

所以定義不做什麼，反而是 ai 時代下，更需要做的事情

P5
然後是，acceptance criteria 

這個也滿重要的，這個是驗證功能的欄位，

就是，你需要去定義，這個功能怎樣才算完成，

這其實也是給 AI 一個邊界，讓他們不要做超過範圍的事情



-------
P6
最後一個欄位是，notes 

這個欄位是用在，功能可能有特殊設計的時候，

就可以寫在這個欄位中

S18
然後，spec 在寫的時候，

要特別注意一件事情，

就是不要把複數功能，混合在一起

因為這樣後續在維護或擴充功能的時候，會比較麻煩

因此，建議每個功能，都有一個獨立的 spec 結構，

這樣後續處理會比較方便

S18
這是實作的流程，

接下來，我們會一直重複跑這個流程，

來把我們的遊戲原型完成


那我們就實際操作給大家看




建立 docs/ 資料夾，然後把 遊戲設計文件 和 storyboard 放到文件中，作為開發遊戲的依據

打開 unity 建立新專案，要把專案資料夾選在跟 codex 的資料夾相同位置，這樣讓 codex 可以讀寫資料

ps: 獨立對話串 (處理雜務)

第一，請 codex 建立 agent.md 文件，這時候 codex 會分析你的專案內容，但因為你的專案目前沒東西，只有遊戲設計文件，所以文件應該只會大略說明而已

這個 agent.md 文件 非常建議定期更新一下，因為每次 codex 在做東西之前，都會讀它。

所以我們可以通過定期更新文件，讓 ai 知道我們目前專案的情況，因為 ai 是沒有記憶的，所以我們要通過這種方式，讓他知道。

ps: 獨立對話串 (專門更新 agent )

第二，把 遊戲設計文件 和 storyboard  餵給 codex ，請 codex 產生一份實作計畫。

可以使用這個提示詞，如下: 參考實作計畫提示詞

最多人遇到的問題是，已經有遊戲想法了，但不知道要從哪裡開始做，這時候就可以請ai幫你規劃，然後你在自己確認一下，這份計畫可不可以，如果你覺得有問題的話，可以跟 ai 討論看要修改哪裡

建議一開始，以可以動，可以玩為標準，不要一開始就要把遊戲完成，因為遊戲的目的是要驗證好不好玩

把實作計畫，寫入到 docs/ 資料夾裡面

ps: 獨立對話串 (專門更新實作計畫)

第三，按照實作計畫書的項目，逐一開始實作，這邊我們需要開始寫spec，那我們有幫大家整理一個提示詞，大家可以直接貼上使用

loop 目標(計畫書中標示的任務) 邊界(建立的spec, plan) 開始跑 驗收(分為ai驗收, 人類驗收(如建立物件和掛腳本等等)) 最終commit 

ps: 感覺要搭 commit ? 不然感覺會爆炸

然後，因為這整個做完的時間太長，所以我們接下這段會跳過，會跳到我們做完的遊戲，然後展示給大家看

我的想法是，會按照實作計畫的id，逐一開始寫spec, plan，以及實作功能，然後等ai和人類驗證，確認完成後，就去改實作計劃的stauts，然後commit。之後開始實作下一個功能。
