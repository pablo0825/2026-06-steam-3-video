# 「程式實作」分鏡文件設計規格

日期：2026-06-22

## 目標

根據以下素材建立「程式實作」章節的完整分鏡腳本：

- `docs/03-程式實作/逐字稿.txt`
- `docs/03-程式實作/briefing-img/`
- 前兩章既有分鏡文件

成品需延續前兩章的文件結構與影片風格，並能直接作為後續 Remotion 動畫與剪輯工作的實作依據。

## 輸出文件

預定建立：

`docs/03-程式實作/steam - 3 - Vibe Game 教案 - 程式實作 - 分鏡腳本.md`

文件包含：

1. 影片基本資訊與視覺規範
2. 頁面／場景製作進度表
3. 各場景的旁白、秒數、畫面、呈現方式與動態
4. 螢幕錄影及外部素材備註
5. 製作前檢查項目
6. 純旁白稿

## 內容範圍

| 場景 | 主題 | 主要呈現方式 |
|---|---|---|
| S01–S03 | 開場、本次重點、相關知識 | 標題動畫、重點卡片、知識清單 |
| S04–S07 | User Story | 定義版型、格式拆解、Rhythm Doctor 影片與範例 |
| S08–S13 | Context 與 AGENTS.md | 概念動畫、書櫃比喻、規則文件及操作示範 |

| S23 | 基本功的重要性 | 重點句逐項進場，不新增旁白 |
| S24 | 結尾 | 感謝文字與收尾動畫 |

## 旁白與文字原則

- 逐字稿是 S01–S18 的主要旁白來源。
- 僅修正明顯錯字、漏字與口語上不通順之處，不改變原意。
- `AGENT.md` 統一修正為 `AGENTS.md`。
- 簡報第 20–22 頁屬於實際操作內容，不依簡報文字擴寫旁白。
- 逐字稿未涵蓋的後段頁面不自行補寫旁白；分鏡只描述畫面內容、操作目的與剪輯方式。
- 純旁白稿只收錄確定存在的旁白，不把畫面指示混入旁白。

## 視覺方向

- 延續前兩章的米白背景、深灰文字與黃色重點色。
- 定義頁使用「標題＋精簡定義＋提示」的統一版型。
- 抽象概念以 SVG 小圖示、卡片、連線與箭頭呈現。
- 清單與流程採逐項進場，避免整頁內容同時出現。
- Rhythm Doctor 與 Celeste 案例優先採滿版影片，再以文字或資訊卡補充。
- 實際操作段落以螢幕錄影為主，動畫僅負責段落標題、操作提示與轉場。

## 場景設計

### S01–S03：建立章節脈絡

- S01 沿用系列開場識別，將章節名稱改為「程式實作」。
- S02 以兩張重點卡片呈現 User Story 與 Spec 驅動開發。
- S03 依序顯示 User Story、Context、AGENTS.md、Spec，建立後續內容索引。

### S04–S07：User Story

- S04 使用統一定義版型說明 User Story 與共同理解。
- S05 將「角色、需求、價值」拆成三個可辨識區塊。
- S06 使用 Rhythm Doctor 影片素材，讓觀眾先觀察遊戲行為。
- S07 以逐欄填入方式呈現完整 User Story 範例。

### S08–S13：Context 與 AGENTS.md

- S08 將 Context 視覺化為 AI 的短期記憶。
- S09 透過對話內容逐漸累積、舊資訊淡出的方式呈現限制。
- S10 使用只能放入固定數量書籍的書櫃比喻 Context 容量。
- S11 說明 AGENTS.md 是每次協作都會參考的規則文件。
- S12–S13 以實際操作錄影呈現建立規則及觀察結果；分鏡記錄操作步驟，不虛構操作結果。

### S14–S18：Spec

- S14 使用定義版型呈現「壓縮 Context」與「維護依據」兩個功能。
- S15 以六張卡片依序介紹 User Story、Input/Output、Rules、Non-goals、Acceptance Criteria、Notes。
- S16 使用 Celeste 影片素材，讓觀眾先觀察跳躍功能。
- S17 以欄位逐項填入的方式呈現跳躍 Spec。
- S18 先顯示「一個功能、一份 Spec」，再轉入 User Story → Spec → Plan → 實作 → 驗證流程。

### S19–S21：實際操作

- S19：建立 Unity 專案、Codex 專案聊天串、AGENTS.md 與 Spec 資料夾。
- S20：整理 User Story、撰寫 Spec、產出 Plan，請 AI 依文件實作功能。
- S21：依 Plan 的驗收項目進行人工驗證。
- 三段均使用實際螢幕錄影，文件只規定必須錄到的操作節點、可刪除的等待時間及必要的畫面放大提示。
- 不替操作段落補寫固定旁白，保留現場講解與後製調整空間。

### S22–S24：補充與結尾

- S22 將「不要刪除舊 Spec／Plan、改建新文件並標記取代關係」拆成兩個步驟。
- S23 以兩段重點句強調程式與 Unity 基本功仍然重要。
- S24 延續系列結尾版型，顯示「感謝各位聆聽」與「祝大家開發順利」。

## 文件撰寫規則

- 場景編號採 `S01` 至 `S24`。
- 對應的 Remotion composition 命名預留為 `Ch3-PageN-*`。
- 秒數以旁白長度與閱讀時間估算，實際操作段落標示為依錄影長度調整。
- 每個場景必須明確記錄：
  - 畫面構成
  - 元素進場順序
  - 元素退場方式
  - 是否需要影片、圖片、SVG 或螢幕錄影
- 不直接沿用簡報截圖作為最終動畫畫面；簡報只作為內容和構圖參考。

## 驗收標準

- 涵蓋逐字稿全部 18 頁及簡報後段的實作、補充與結尾內容。
- 前後場景編號連續，且旁白與簡報主題沒有錯置。
- 所有 `AGENT.md` 均統一為 `AGENTS.md`。
- S19–S21 明確標示為實際操作，沒有自行補寫旁白。
- 視覺、動態與文件格式可與前兩章自然銜接。
- 每個場景都具備足以開始實作或剪輯的畫面與動態描述。
